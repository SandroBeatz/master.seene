# i18n Text Guard — Skill Specification

> Spec v1.0 · 2026-07-08 · status: built · [Skills index](./README.md)

## 1. Purpose

This skill ensures that no human-readable string is ever hardcoded in the Seene UI. The project ships in three locales — English (default), French, Russian — via vue-i18n, and any literal label, title, placeholder, `aria-label`, or `alt` that bypasses the translation layer is a functional bug: it renders untranslated for two-thirds of users.

The skill enforces the rule at write-time. Before any markup or component work that produces visible text, it directs the agent to route that text through `$t()` / `t()`, reuse existing translation keys when the phrase already exists, scope new keys correctly, and add every new key to all three locale files at once. In `--check` mode it scans the project for existing hardcoded strings and migrates them interactively.

Its value: translations stay complete and consistent by construction, instead of drifting as untranslated literals accumulate.

## 2. When to trigger

**Should trigger:**
- Any UI/markup/component work involving visible text: writing or editing `.vue` templates, adding text props (labels, titles, descriptions, placeholders), building pages, modifying layouts, nav items, error messages, `aria-label`/`alt` text.
- Even minimal-sounding requests: "add a button", "update the title", "change this label".
- The explicit command `/i18n --check` (or "find hardcoded strings", "audit translations").

**Should NOT trigger:**
- Changes with no human-readable string: pure logic, styling classes, icon names (`i-lucide-*`), route paths, URLs, config values, numeric/boolean props.
- Backend/database/migration work.
- Pure component-API questions (defer to `[[nuxt-ui]]`) or file-placement questions (defer to `[[fsd]]`) — though any visible text those tasks introduce still routes through this skill.

## 3. Inputs

- **User request** describing UI work, or the `--check` flag.
- **Locale files** `src/shared/lib/i18n/locales/{en,fr,ru}.ts` — read to check for an existing equivalent key before creating a new one, and written to when adding keys.
- **The `.vue` files under `src/`** — read in `--check` mode to find hardcoded strings.
- **Locale store** `src/stores/locale.ts` (`useLocaleStore()` with `current` + `setLocale`) for locale-switching UI.

## 4. Outputs

**Guard mode** — no fixed template. Visible text is emitted as `$t('key')` in templates or `t('key')` in `<script setup>` (with `useI18n` imported and `t` destructured; reactive arrays/objects wrapped in `computed()`), and the corresponding key is added to `en.ts`, `fr.ts`, and `ru.ts`.

**Audit mode (`--check`)** — findings grouped by file (file path, line number, the hardcoded string, a suggested key name), then a prompt: `"Found N hardcoded strings. Translate them? (yes / no / select)"`, followed by the applied migrations for approved items.

## 5. Workflow

**Guard mode — for each piece of visible text:**

1. **Detect visible text** in the markup being written/edited — labels, titles, descriptions, buttons, placeholders, nav items, error messages, and the `aria-label` / `alt` attributes (these are text too).
2. **Reuse before creating.** Read `src/shared/lib/i18n/locales/en.ts`; if an equivalent phrase already exists, use that key. Never duplicate a phrase under a new key.
3. **Scope a new key by context** (camelCase, descriptive, grouped under a shared parent):
   - Global layout / sidebar nav → `nav.{name}`
   - Page header/title → `{page}.title`
   - Page section → `{page}.{section}.{name}`
   - Shared reusable UI → `common.{name}`
   - Component-specific → `{componentName}.{name}`
   Avoid generic keys like `button`/`text` that break across contexts.
4. **Add the key to all three locale files in the same change.** Never leave a locale missing a key — vue-i18n falls back to `en` silently, masking the gap. If the FR/RU translation is unknown, provide a best-effort one; a reasonable translation beats a missing key (which shows the raw key string in non-EN locales).
5. **Emit the reference correctly:** `$t('key')` in `<template>` (globally available, no import); `t('key')` in `<script setup>` (import `useI18n`, destructure `t`). Wrap text in `computed()` when it lives in a reactive array/object (nav items, table columns, tabs) so it re-renders on locale switch; plain template interpolation is already reactive and needs no wrapper.
6. **Do not locale-prefix routes.** URLs are the same across languages; plain Vue Router paths (`to="/settings"`) are correct. Never use `$localePath()` — it doesn't exist in this project.

**Audit mode (`--check`):**

1. **Scan** all `src/**/*.vue` for hardcoded human-readable strings: plain-string attribute values that aren't keys/paths/URLs/icons (e.g. `label="Settings"`) and template text nodes (e.g. `<h2>General</h2>`). Skip icon names, URLs, route paths, CSS classes, color names, and anything already using `$t()`/`t()`.
2. **Present findings** grouped by file with line numbers, the offending string, and a suggested key following the §5.3 scoping rules.
3. **Ask** `"Found N hardcoded strings. Translate them? (yes / no / select)"` — `yes` migrates all, `no` does nothing, `select` migrates only named files/strings.
4. **Migrate** each approved string: replace with `$t('key')` (template) or `t('key')` (script, ensuring `useI18n`/`t` present), add the key to all three locales. If the source string is Russian, use it as the `ru` value and translate to `en`/`fr`; if English, use it as `en` and translate to `fr`/`ru`.

## 6. Resources

No bundled scripts. The skill needs only knowledge of the three fixed locale-file paths, the key-scoping table, and the `computed()` reactivity rule — all self-contained in the skill body. A materializer should preserve the exact locale-file paths and the "update all three locales simultaneously" rule, as correctness depends on them.

## 7. Examples

**Example 1 — guard:**
- Input: "Add a Save button to the settings form."
- Output: `<UButton>{{ $t('common.save') }}</UButton>` (reusing `common.save` if present, else adding it to `en.ts`/`fr.ts`/`ru.ts`).

**Example 2 — audit:**
- Input: `/i18n --check`
- Output: grouped findings such as `src/pages/settings/ui/SettingsPage.vue:42 — "General" → settings.general.title`, the prompt `Found 3 hardcoded strings. Translate them? (yes / no / select)`, then the applied edits after the user answers.

## 8. Acceptance criteria

- No visible text the built skill writes is a bare literal; all of it is `$t()`/`t()`.
- Every new key exists in all three of `en.ts`, `fr.ts`, `ru.ts` — never a subset.
- Existing equivalent keys are reused rather than duplicated (skill reads `en.ts` first).
- Reactive-context text (arrays/objects in `<script setup>`) is wrapped in `computed()`.
- `aria-label` and `alt` are translated; routes are not locale-prefixed and `$localePath()` is never introduced.
- `--check` groups findings by file with line numbers and suggested keys, prompts with the exact `yes / no / select` question, and honors the answer.
