# Seene — Roadmap до теста у мастеров

**Дата составления:** 2026-06-26
**Цель:** TestFlight-сборка iOS + рабочая публичная часть, отправка мастерам (Канада, Бишкек)
**Окно теста:** ~24 авг – 7 сен 2026
**Ресурс:** соло (Сандро + ассистент)
**Платформа теста:** нативный iOS через Capacitor → раздача через **TestFlight** (не публичный App Store)

---

## Главный вывод

23 задачи из исходного списка соло за 10 недель целиком не закрыть. Нативный iOS — длинный хвост (адаптив → Capacitor → push → ревью), он диктует критический путь. Поэтому:

- **MUST для теста** — то, без чего мастер не сможет реально вести записи и дать осмысленный фидбек.
- **DEFER (после первого теста)** — Портфолио, мультисайз картинок, Activity-блок, публичный релиз в App Store. Это улучшения, не блокеры.

Если по ходу горит дата — режем сначала из DEFER, потом из Phase 2 (лендинг-косметика), никогда из критического пути (auth → mobile → Capacitor → TestFlight).

## Критический путь (то, что нельзя начать поздно)

```
Apple Developer аккаунт ──┐
Реальный OAuth (Apple req)─┼─► Адаптив/mobile web ─► Capacitor+Ionic ─► Push ─► TestFlight ─► Мастера
Privacy/Terms (App Store) ─┘
```

Apple-аккаунт и OAuth — **завести в неделю 1**, у них внешний лид-тайм (верификация, провижининг).

## План по фазам

### Phase 0 — Фундамент и разблокировка · Неделя 1 (26 июн – 5 июл)
Внешние зависимости с лид-таймом + то, что блокирует остальное.

1. **Apple Developer аккаунт** — оформить сегодня ($99/год, верификация занимает дни). Без него нет TestFlight.
2. **Задача 12 — реальный Google/Apple OAuth** в Supabase (не заглушка). Apple требует Sign in with Apple, если есть любой сторонний вход — это блокер ревью. UI-кнопки уже стоят.
3. **Landing задача 1 — Privacy Policy + Terms** с реальным текстом. Требуется для submission в App Store и для Supabase. Сейчас заглушки.
4. ~~**Гигиена Supabase** — закрыть security warnings (advisors), перепроверить RLS после booking-миграций.~~ ✅ Сделано 2026-07-08 (`supabase/migrations/20260708120000_supabase_security_hygiene.sql`): удалена мёртвая RPC `find_client_name_by_phone` (SECURITY DEFINER без проверки владельца — давала перебор клиентов чужих мастеров), закрыт анонимный доступ к `complete_appointment` и `rls_auto_enable`, закреплён `search_path` у функций, заодно закрыты performance-advisories (`auth_rls_initplan`, `multiple_permissive_policies`). Вручную осталось включить Leaked Password Protection в Dashboard → Authentication → Settings.

### Phase 1 — Ядро дашборда · Недели 2–3 (6–19 июл)
Ежедневные действия мастера — без них тест бессмысленный.

5. **Задача 11 — быстрое создание записи с главной** (сервис → слот → клиент за 20–30 сек) + создание Time off. Сейчас только через календарь. Это главная ежедневная операция.
6. **Задача 2 — уведомления (web)**: настройка в Supabase + доставка для веба. Форма настроек уже есть.
7. **Задача 14a — клиентские WhatsApp-шаблоны** (мастер выбирает тему → отправляет заготовленный текст вручную). Прямо связано с уведомлениями, ядро ценности продукта.
8. **Задача 10 — эмуляция подписки**: поля в БД, смена плана без оплаты, определить фичи по тарифам, валидация (гейтинг). UI-заглушка уже есть, нужна логика.

### Phase 2 — Аналитика + публичная часть · Недели 3–4 (13–26 июл, частично параллельно Phase 1)
9. **Задача 1 (analytics) — доработка** по дизайну + доп. блоки. База уже есть, это не с нуля.
10. **Landing 2,3,4 — About, 404, Contacts** (включая выбор каналов связи). Нужны и для доверия, и для ревью Apple (support URL).
11. **Landing 5 — Home UI лендинга**: актуальный контент по секциям + кнопки на приложения.
12. **Landing 6 — Master page**: обновить UI, кешировать данные юзера, отрендерить все нужные данные мастера.
13. **Landing 7 — Online booking полировка**: прелоадеры, больше инфо о встрече и отмене, кеширование данных при букинге.

### Phase 3 — Мобильный web · Недели 4–6 (20 июл – 9 авг) — длинный хвост, не затягивать
14. **Задача 7 — адаптив + мобильная навигация**: промежуточные страницы, роутинг. Обязательно ДО Capacitor.
15. **Задача 13 — онбординг**: фиксы UI + флоу + мобильная версия.
16. **Задача 6 — UI-фиксы pass**: по списку пройтись по каждой странице и модалке (тост, кнопки, поля, модалки), сразу под мобайл.

### Phase 4 — Нативная упаковка · Недели 6–8 (3–23 авг)
17. **Задача 8 — Capacitor + Ionic** (Ionic только для навигации). Сборка iOS-проекта в Xcode.
18. **Задача 9 — push-уведомления** через Capacitor (iOS приоритет).
19. **Задача 15 — внешняя аналитика** (PostHog/аналог). Поставить ДО теста, чтобы собирать поведение мастеров.

### Phase 5 — Релиз для теста · Недели 8–9 (17 авг – 7 сен)
20. **TestFlight**: сертификаты, провижининг, загрузка билда, internal/external testers. (Усечённая версия задачи 14b — публичный App Store потом.)
21. **Финальный QA**: прогон всех флоу на реальном iPhone, верификация end-to-end (онбординг → услуги → клиентский букинг → мастер видит → создаёт запись → шлёт уведомление → аналитика).
22. **Задача 16 — отправка мастерам** (Канада, Бишкек) + поиск нескольких в Threads.

## DEFER — после первого теста

- **Задача 3 — Портфолио** (определиться: работы в сервисах vs отдельный раздел с привязкой к сервису). Не блокер букинг-теста.
- **Задача 4 — мультисайз картинок** в Supabase. Аватар уже ресайзится; полная оптимизация имеет смысл только после портфолио.
- **Задача 5 — Activity-блок** на главной. Nice-to-have виджет.
- **Задача 14b — публичный релиз в App Store**. Для теста достаточно TestFlight; полный релиз после фидбека.
- **Чистка legacy booking-пути** (найдено при Supabase-гигиене 2026-07-08): таблицы `bookings`/`availability` и RPC `create_booking` заменены на `appointments`/`create_appointment_from_booking`, не пишутся (0 строк), но `bookings` всё ещё читается defensively в `seene/server/api/master/[username]/slots.get.ts` и `send.post.ts`. Кросс-репо задача (`seene` + `master.seene`), не блокер теста.

## Риски

- **Длинный хвост iOS.** Если Phase 3 (mobile web) уедет за неделю 6 — Capacitor и TestFlight не успевают. Mobile web — самый контролируемый рычаг даты.
- **Ревью Apple.** Sign in with Apple, Privacy/Terms, support-контакт — частые причины реджекта. Поэтому они в Phase 0.
- **Объём соло.** Если к концу июля Phase 0–1 не закрыты — резать Phase 2 (лендинг-косметику) и оставить минимум: About/404/Contacts + booking-полировка.
